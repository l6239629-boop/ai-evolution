# 迭代开发工作流

四种协作模式覆盖日常开发全场景。每种模式的核心原则：**先沟通确认方向，再动手执行——永远不要在模糊需求上浪费 token。**

## 模式速查

| 模式 | 适用场景 | 流程 |
|------|---------|------|
| 规划确认 | 新功能、新模块 | 产品方案→确认→技术方案→确认→执行 |
| 分步推进 | 需调研外部资料 | 调研→理解总结→方案→审核→技术方案→确认→执行 |
| 对比定位 | 优化、修复、重构 | 理解现状→对照目标找问题→制定修改→审核→确认→执行 |
| 回退锚点 | 关键节点 | 备份当前状态→版本号递增→继续迭代 |

## 模式 1：规划确认

**何时用**：新功能开发、模块新增。需求相对明确，不需要外部资料调研。

**Prompt 模板**（追加到任务末尾）：

```
请你先规划这个功能需求如何迭代制定，产出产品方案。然后跟我确认。我确定后再制定技术实现方案。
```

**流程**：
1. 出产品方案（做什么、长什么样）
2. 用户确认
3. 出技术方案（怎么改、改哪里）
4. 用户确认
5. 执行

## 模式 2：分步推进

**何时用**：需要调研外部链接/文档/资料后才能推进的任务。

**Prompt 模板**：

```
请你先阅读链接里面的内容，然后进行理解总结。制定一个讲解阐述方案。然后跟我审核。通过后制定技术调整方案。
```

**流程**：
1. 阅读外部资料，总结
2. 出讲解/阐述方案
3. 用户审核
4. 出技术调整方案
5. 用户确认
6. 执行

## 模式 3：对比定位

**何时用**：优化现有功能、修复 bug、对照目标找出差距。

**Prompt 模板**：

```
请你先理解以上内容，然后对照现状找出问题以及要补充的内容，制定修改内容。然后先跟我审核。确认后才可执行。
```

**流程**：
1. 理解当前状态和期望目标
2. 逐项对比，找出差异点和缺失点
3. 制定修改清单
4. 用户审核确认
5. 执行每条修改

## 模式 4：回退锚点

**何时用**：每个确认节点后、每次大改动前、每天收工时。

**Prompt 模板**：

```
请你对以上内容进行备份，按照 v1 命名的格式按顺序储存迭代。
```

**版本命名规范**：
- 主版本：`v1.0` → `v2` → `v3` → ...（简洁递增）
- 备份文件：`index-vN.html` 或 `project-vN.ext`
- Git tag：`vN`（与备份文件同步）
- 每次大迭代前，先执行备份

## 版本管理三件套

### 1. 备份文件

```bash
cp index.html index-v7.html
```

### 2. Git 标签

```bash
git add . && git commit -m "v7: 变更描述" && git tag v7
```

### 3. CHANGELOG 自动更新

使用 `update-changelog.sh`（见 assets/）：

```bash
bash update-changelog.sh v7 "变更标题" "新增A;新增B" "修改C" "修复D"
```

参数：版本号 + 标题（必填），新增/修改/修复用分号 `;` 分隔多条。

CHANGELOG.md 格式规范见 `references/changelog.md`。

## 注意事项

- **四个模式可组合**：一个任务可能先用模式 2 调研，再用模式 1 规划，最后用模式 4 备份
- **确认环节不可跳过**：用户确认是整个流程的安全网，防止方向错误造成大量返工
- **备份越频繁越好**：每次确认通过后立即备份，确保任何时刻可回退
